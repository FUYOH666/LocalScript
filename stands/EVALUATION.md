# Оценка продукта по стенду E2E

Документ основан на реальном прогоне `stands/run_e2e.py` (мок LLM + HTTP + опционально Docker `luac`).

## Что проверяет стенд

1. **Мок vLLM/OpenAI** (`stands/mock_llm.py`): `GET /v1/models`, `POST /v1/chat/completions`.
2. **LocalScript**: `GET /healthz`, `POST /generate`.
3. **Фаза happy**: модель сразу отдаёт валидный Lua — ожидается `success: true`.
4. **Фаза fix_loop**: первая выдача — синтаксически битый Lua; вторая и далее — валидный. Ожидается **несколько шагов** агента, если на хосте реально работают линтеры/`luac`.
5. **Docker luac** (если есть Docker): независимая проверка, что текст в `code` — валидный Lua.

## Наблюдения с прогона (типичный Mac без brew Lua)

Пример фактического вывода:

- **Инструменты на хосте**: все четыре (`stylua`, `selene`, `lua-language-server`, `luac`) могут быть **не установлены** — тогда шаги валидации **пропускаются** как «skipped», цепочка считается успешной без реальной проверки кода.
- **healthz**: `llm_ok: true`, `validators_ready: false`, `status: degraded` — это честный сигнал: LLM доступен, но **контур валидации не готов**.
- **generate_happy**: `success: true`, `steps: 1`; **docker_luac: pass** подтверждает, что сгенерированный текст действительно Lua.
- **generate_fix_loop**: при отсутствии валидаторов возможен **`success: true` за один шаг** — это **ложноположительный** сценарий: битый первый ответ мока не был отловлен, потому что нечем проверить.

Вывод: продукт **корректно живёт без тулов** (удобно для демо), но **гарантии качества в таком режиме нет** — это нужно явно мониторить и, по возможности, запрещать для продакшена.

## Субъективная оценка (инженерная)

| Критерий | Оценка | Комментарий |
|----------|--------|-------------|
| Контракт API | 4/5 | Понятные `healthz` / `generate`, не хватает idempotency-key / trace id в ответе |
| Наблюдаемость | 3/5 | Логи есть; в ответе `steps` полезны, но нет correlation id сквозь LLM и линтеры |
| Безопасность доверия к коду | 2/5 без тулов, 4/5 с полным пайплайном | Без линтеров/`luac` система **не отличает** хороший и плохой Lua |
| Воспроизводимость стенда | 4/5 | Мок + скрипт стабильны; Docker зависит от сети на первый pull образа |
| Соответствие заявленной архитектуре | 4/5 | Цепочка StyLua→Selene→LuaLS→`luac` задумана верно; LuaLS CLI может отличаться по версиям |

## Идеи улучшения (приоритет сверху)

1. **Режим «строгой готовности»** (`LOCALSCRIPT_REQUIRE_VALIDATORS=true`): если ни один инструмент не доступен — `POST /generate` возвращает **503** с понятным телом, либо только `healthz` остаётся 200 с `status: degraded` (уже есть `validators_ready`).
2. **Отдельный флаг «syntax-only минимум»**: разрешить работу только с `luac` (или только Docker-раннером), чтобы на чистом Mac был минимальный барьер без полного набора линтеров.
3. **В ответе `/generate`**: поле `validation_profile: skipped|partial|full` и список фактически выполненных шагов (не только raw в шагах).
4. **Сквозной `request_id`**: принять заголовок `X-Request-ID`, прокинуть в логи и в каждый `AgentStepLog` в JSON.
5. **Стриминг** (опционально): `POST /generate` с SSE для длинных задач и отладки в UI.
6. **Контрактные тесты против реального vLLM**: отдельный job (не в CI по умолчанию) с секретом `BASE_URL` в периметре.
7. **Песочница выполнения** (не только `luac -p`): Docker/nsjail с `load(..., "t", env)` и заглушками API, как в compass-документе.

## Как интерпретировать стенд для демо

- Для **честного** демо установите хотя бы **`luac`** или подключите **`LOCALSCRIPT_PATH_EXTRA`** к brew-бинарям и убедитесь, что `validators_ready: true`.
- Для **быстрой** проверки интеграции LLM достаточно мока + `docker luac` в `run_e2e.py`.
